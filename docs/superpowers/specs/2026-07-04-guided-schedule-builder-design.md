# Guided Schedule Builder — Design Spec

**Date:** 2026-07-04
**Status:** Approved by user, pending spec review

## Context

The current Schedule section of Add/Edit Medication (`ScheduleBuilderView.swift`) is a flat form: a time picker, a Daily/Weekly/Custom frequency segmented control, and (for non-daily) a day-of-week toggle row. Adding more than one dose per day means manually adding another `ScheduleDraft` row and independently setting its time — there's no guidance connecting "I take this 3 times a day" to the individual times that implies, and no support at all for scheduling relative to meals (common for antibiotics and other meal-timed medications).

This spec replaces that flat form with a guided, question-driven flow for anything beyond the simplest case, while keeping a fast single-line default for the common "once a day, every day" medication so the common case isn't penalized by extra taps.

## Goals

- A collapsed, compact default ("Taken: Once daily at [time]") for the common case — no guided flow shown unless the user changes something.
- A step-by-step question sequence for anything more complex (specific days, multiple times per day), asking one thing at a time rather than presenting a form with several unrelated controls at once.
- Support scheduling doses relative to meals (Breakfast, Morning Tea, Lunch, Afternoon Tea, Dinner, Dessert, Midnight Snack) as an alternative to fixed clock times, since this is a common real-world pattern (e.g. "take with meals").
- A single, app-wide set of meal times (not per-medication) that the user can set inline the first time they use a meal-tied schedule, or later via Settings → Preferences → Meal Times.
- All schedule data generated by this flow, and the meal-times preference itself, must sync to Supabase so it survives app reinstalls, device changes, and crashes — consistent with how the rest of the app's data already works.

## Non-goals

- No changes to the underlying `Schedule`/`ScheduleDraft` Core Data model or its Supabase table — the guided flow only changes *how* `hour`/`minute`/`daysOfWeek` values get filled in, not what's stored.
- No per-medication override of meal times — meal times are global, matching how meals actually work for a person regardless of which medication is being scheduled (per user: "the person is the variable with meal times, not the medication").
- No automatic interval calculation from "times per day" (e.g. dividing 24 by N) — the app never assumes doses are evenly spread across a full day, since that ignores sleep hours. Interval-based generation always starts from a user-supplied first-dose time and a user-supplied hour interval.

## UI Flow

**Entry point:** The Schedule section defaults to a single collapsed row summarizing the current schedule (e.g. "Taken: Once daily at 8:00 AM"). Tapping it expands into the guided flow below. A brand-new medication starts pre-expanded on Question 1 (nothing to summarize yet, and iOS is 17+, so pushing to guided-flow-first is fine for a new medication — the fast path exists once a schedule already exists, when *editing*, to spare returning users from question dead ends).

**Question sequence** (one question visible at a time, each answer revealing the next):

1. **"How often is [dose] of [medication name] taken?"** → **Every day** / **Specific days**
   - If Specific days → show the existing day-of-week toggle row (unchanged from today), then continue to Q2.
2. **"How many times per day?"** → numeric stepper (1+).
   - If 1 → single time picker, flow ends, generates one `ScheduleDraft`.
3. **If >1 — "How are doses spaced?"** → **Fixed intervals** / **Tied to meals** / **Set each time manually**
   - **Fixed intervals:**
     - Ask "First dose time" (time picker) and "Hours between doses" (integer, full hours only).
     - Generate N times: `first`, `first + interval`, `first + 2×interval`, ... for `i in 0..<N` — never divides 24 hours by N. Times may land past midnight (e.g. 4 doses/6h from 10pm → 10pm, 4am, 10am, 4pm) — this is expected and not an error case.
   - **Tied to meals:**
     - Show a checklist of the 7 meal slots, each pre-filled with the current global meal time (see below) and independently adjustable right there via a time wheel.
     - The count of meals selected must equal the times-per-day answer from Q2 before continuing (show an inline count, e.g. "2 of 3 selected").
     - Selecting a meal generates a `ScheduleDraft` at that meal's current global time.
   - **Set each time manually:**
     - Show N sequential time pickers, one per dose ("Dose 1 time", "Dose 2 time", ... "Dose N time").
4. **Review step:** before committing, show the full generated list of times (and days, if not every day) as a simple editable summary — this is where an odd interval-wrap result becomes visible and directly fixable, rather than silently applied.

## Meal Times (new global preference)

**Model:** a fixed set of 7 named slots, each holding an hour/minute:

| Slot | Default |
|---|---|
| Breakfast | 7:30 AM |
| Morning Tea | 10:00 AM |
| Lunch | 12:30 PM |
| Afternoon Tea | 3:00 PM |
| Dinner | 6:30 PM |
| Dessert | 7:30 PM |
| Midnight Snack | 11:00 PM |

**Storage:** persisted locally (UserDefaults, alongside the app's other preference keys) and synced through the existing `SupabaseSyncManager` / `user_settings` mechanism — the same round-trip already used for other preferences (time format, haptics, etc.), so it survives reinstall/device-change/crash like everything else does. No new table needed; this is additional columns (or a single encoded blob) on the existing `user_settings` row.

**Editing:** two entry points write to the same shared value —
- Inline, during the "Tied to meals" step of the guided flow (adjust-and-continue, doesn't require a separate trip to Settings first).
- **Settings → Preferences → Meal Times**, a new list screen (7 rows, each opens a time picker) — for changing meal times independent of adding a medication.

## Data Flow / Persistence

- The guided flow's only output is the same `ScheduleDraft` array `AddEditMedicationViewModel` already produces — no new Core Data entity, no new Supabase table for schedules themselves.
- Meal times are the one new piece of state, synced via the existing settings sync path (`SupabaseSyncManager`'s push/pull of `user_settings`), so they reach other devices signed into the same account and survive local data loss.
- As today, saving the medication pushes its `Schedule` rows to Supabase via the existing `pushMedication`/`pushSchedule` flow — this spec doesn't change that mechanism, just what feeds into it.

## Testing

- Unit tests for the interval-generation function: `(first: (hour, minute), intervalHours: Int, count: Int) -> [(hour, minute)]`, covering non-wrapping and midnight-wrapping cases, as a pure function independent of the UI.
- Unit tests confirming meal-time selection generates a `ScheduleDraft` matching the current global meal time at generation time (not a stale/default value).
- Manual verification of the full question sequence for each branch (every day / specific days × fixed interval / meals / manual), and of the Settings → Preferences → Meal Times screen editing a value that a subsequent meal-tied schedule then reflects.
